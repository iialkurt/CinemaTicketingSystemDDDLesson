# Test Failures due possible code bugs

## CinemaTicketingSystem.Application.Test.csproj - CinemaTicketingSystem.Application.Test.TicketIssuanceAppServiceTests.Create_ShouldIgnoreOtherUserSeatsWhenCheckingAvailability
- **Confidence**: High
- **Test File**: CinemaTicketingSystem.Application.Test\TicketIssuanceAppServiceTests.cs
- **Bug Location**: CinemaTicketingSystem.Domain\BoundedContexts\Ticketing\Issuance\TicketIssuance.cs@51

### Analysis
The error 'The property TicketIssuanceId belongs to the type TicketIssuance.TotalPrice#Price, but is being used with an instance of type Ticket.Price#Price' is an Entity Framework Core configuration issue. The problem occurs because both TicketIssuance and Ticket have a Price property (TicketIssuance.TotalPrice and Ticket.Price). EF Core is treating Price as an owned entity type and is getting confused about which navigation/foreign key belongs where. Looking at the TicketIssuance class (line 51), there's a TotalPrice property of type Price that gets set in the CalculateTotalPrice method (lines 102-117). The CalculateTotalPrice method is called when tickets are added (line 81). The problem is that when EF Core tries to persist the TicketIssuance aggregate with its collection of Tickets (each having their own Price) AND the TicketIssuance's TotalPrice, it's incorrectly trying to associate the Ticket.Price owned entity with a TicketIssuanceId foreign key, when that foreign key should only belong to TicketIssuance.TotalPrice. This is a configuration error in the entity mappings where Price needs to be configured differently for different contexts, or the TotalPrice property needs initialization before the entity is added to the context.

### Suggested Fix
The issue is in the TicketIssuance domain class. The TotalPrice property (line 51) is not initialized when a new TicketIssuance is created. When AddTicket is called, CalculateTotalPrice() sets the TotalPrice, but if no tickets are added before persisting, TotalPrice remains null. More critically, the constructor at line 32-39 doesn't initialize TotalPrice, and even after AddTicket is called in the production code (line 103 in TicketIssuanceAppService), EF Core's change tracking may be getting confused.

The fix is to initialize TotalPrice in the TicketIssuance constructor:

```csharp
public TicketIssuance(Guid scheduleId, CustomerId customerId, DateOnly screeningDate)
{
    Id = Guid.CreateVersion7();
    ScheduledMovieShowId = scheduleId;
    CustomerId = customerId;
    ScreeningDate = screeningDate;
    Status = TicketIssuanceStatus.Created;
    TotalPrice = new Price(0, "TRY"); // Initialize with zero price
}
```

Additionally, if there's an EF Core entity configuration file for TicketIssuance, the Price owned entity should be explicitly configured with different table names or columns to avoid conflicts:

```csharp
builder.OwnsOne(x => x.TotalPrice, priceBuilder =>
{
    priceBuilder.Property(p => p.Amount).HasColumnName("TotalPrice_Amount");
    priceBuilder.Property(p => p.Currency).HasColumnName("TotalPrice_Currency");
});

builder.OwnsMany(x => x.TicketList, ticketBuilder =>
{
    ticketBuilder.OwnsOne(t => t.Price, priceBuilder =>
    {
        priceBuilder.Property(p => p.Amount).HasColumnName("Price_Amount");
        priceBuilder.Property(p => p.Currency).HasColumnName("Price_Currency");
    });
});
```
